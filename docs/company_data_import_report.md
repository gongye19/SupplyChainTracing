## 6. 入库过滤逻辑与过滤内容

当前脚本是全量重建，通过 staging 表导入，成功后再切换正式表。过滤逻辑如下：

1. 原始行级过滤
   - 必须有合法 `月度`。
   - 必须能得到进口侧国家：`进口商所在国家`，缺失时用 `目的国`。
   - 必须能得到出口侧国家：`出口商所在国家`，缺失时用 `原产国`。
   - 当前数据在这些关键条件下跳过 0 行。

2. 公司看板公司范围过滤
   - 只保留 215 品牌白名单内的 13,295 个公司名。
   - 不在白名单内的公司不会进入公司级看板相关表。
   - 过滤掉的非白名单公司名：240,655。

3. 公司聚合表过滤结果

| 过滤项 | 过滤行数 |
| --- | ---: |
| company_monthly_trade_stats 因非白名单公司过滤 | 1,246,344 |
| company_hs_trade_stats 因非白名单公司过滤 | 443,479 |
| company_counterparty_trade_stats 因主体公司非白名单过滤 | 761,464 |
| company_counterparty_trade_stats 白名单公司内 Top 25 截断过滤 | 83,489 |
| country_origin_trade_stats 过滤 | 0 |

4. Counterparty Top 25 截断
   - 对每个 `公司名 + 国家 + role`，按 `金额美元`、`交易次数` 排序，只保留前 25 个交易对手。
   - 这是为了控制 Railway Postgres 的表体积和查询速度。
   - 该截断只影响公司看板下方的 Top Suppliers / Customers，不影响公司自身月度趋势、HS 分类，也不影响前三个国家/HS 看板。

5. 国家/HS 看板数据
   - `country_origin_trade_stats` 使用全部通过行级校验的数据。
   - 不受 215 品牌白名单过滤影响。
   - 因此前三个看板仍是全量国家对贸易聚合数据。

## 7. 结论

当前数据质量整体可用：

- 原始 2,299,728 行全部通过导入脚本关键校验，没有月份、关键国家、公司名、金额空值、交易次数空值导致的跳过。
- 215 品牌白名单完整，覆盖 13,295 个唯一公司名。
- 公司看板最终按 `公司名 + 国家` 拆成 25,430 个公司实体，其中 3,142 个公司名对应多个国家。
- 公司看板不是全量公司数据，而是 215 品牌白名单公司数据；非白名单公司不会作为主体公司进入第四看板。
- 前三个看板使用 `country_origin_trade_stats`，仍保留全量国家/HS 聚合，不受公司白名单过滤。
- 唯一需要注意的是 7,301 行金额为 0 的记录当前被保留；如果业务上认为 0 金额不可用，需要额外增加过滤规则。
